# 站点适配维护

四个适配器位于 `src/providers`。每个适配器只定义三类选择器：编辑器、发送按钮、用户消息；通用写入和验证逻辑位于 `deliver.ts`。

## 页面变化后的排查顺序

1. 在浏览器开发者工具中确认编辑器是 `textarea`、`input` 还是 `contenteditable`。
2. 优先寻找稳定语义属性，例如 `role="textbox"`、`aria-label`、`data-testid` 和消息作者角色。
3. 避免散列类名、DOM 绝对路径、屏幕坐标和仅依赖本地化按钮文字。
4. 更新对应适配器的选择器数组，保持最具体的选择器在前。
5. 在 `tests/providers.test.ts` 更新 DOM fixture，并运行 `npm run check`。
6. 先用“仅填入”模式人工验证，再使用一个无敏感内容的测试词验证自动发送。

## 成功判定

自动发送只有两条成功路径：

- 用户消息节点数量增加，且最新消息包含当前单词或提示词标记；
- 输入框被页面清空。

如果内容明确仍在输入框，返回 `not_submitted`；如果页面发生变化但无法证明消息出现，返回 `submission_unknown`。后者禁止改投，以免重复发送。

## 现场诊断建议

- 检查内容脚本是否已在扩展设置中获得该域名权限。
- 检查站点是否处于登录墙、验证码、附件弹层或历史只读页面。
- 检查按钮是否存在 `disabled` 或 `aria-disabled="true"`。
- 不要通过读取聊天历史或网络请求体绕过页面验证。
